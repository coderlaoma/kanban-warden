# kanban-warden v0.4 Loop Supervisor Self-Improvement Design

## Purpose

`kanban-warden` v0.4 extends the Loop Supervisor from task supervision into controlled self-improvement. The goal is not to let a production gateway silently rewrite itself. The goal is to let the supervisor collect evidence from loop traces, identify weak rules or missing detectors, prepare a bounded improvement, and route that improvement through human approval, verification, audit, commit, push, deployment, and rollback.

This document preserves the v0.4 design so earlier v0.2/v0.3 work can build toward it without losing the target shape.

## Positioning

v0.2 should make task loops observable and verifiable:

- loop state
- loop trace
- post-action verification
- no-progress detection

v0.3 should make policy improvement possible:

- improvement signals from loop outcomes
- policy/config proposals
- approval and audit trail
- safe config apply path

v0.4 adds code-level self-improvement:

- generate code-change proposals when policy cannot express the needed behavior
- create isolated implementation branches
- patch code, tests, docs, and examples
- run verification gates
- request human approval
- push branch or create MR
- deploy/update after approval
- monitor post-release and support rollback

## Non-Goals

- No production process directly edits its own running source code.
- No unapproved code changes are applied to live Hermes profiles.
- No hidden mutation of Kanban data, warden state DBs, or plugin installs.
- No automatic merge to a protected branch.
- No automatic rollout without an auditable approval record.
- No broad refactors bundled with a narrow detector or policy improvement.

## Improvement Levels

| Level | Name | Behavior |
|---|---|---|
| E0 | Observe | Record loop traces, outcomes, false positives, overrides, and verification failures. |
| E1 | Recommend | Produce improvement suggestions with evidence, but do not modify files. |
| E2 | Config Change | Prepare config or policy patches and apply only after approval. |
| E3 | Code Change | Prepare code, tests, docs, and a branch/MR after approval. |
| E4 | Controlled Rollout | After approval and verification, deploy/update the target plugin/profile and monitor. |

v0.4 covers E3 and E4. It depends on v0.2/v0.3 evidence and proposal data.

## Data Model

### loop_trace

Records what the supervisor observed and decided for a task loop.

Fields:

- `trace_id`
- `task_id`
- `board_id`
- `profile`
- `loop_state`
- `observed_facts_json`
- `matched_policy`
- `decision`
- `confidence`
- `planned_action_json`
- `verification_contract_json`
- `created_at`

### loop_outcome

Records what happened after a decision or action.

Fields:

- `outcome_id`
- `trace_id`
- `task_id`
- `action_type`
- `status`
- `verification_status`
- `human_override`
- `override_reason`
- `latency_seconds`
- `created_at`

### improvement_signal

Aggregates repeated evidence that the supervisor's current behavior can improve.

Examples:

- repeated false-positive stale alerts
- repeated no-progress misses
- repeated verification failures after the same action type
- repeated human overrides of the same policy
- repeated worker failure pattern not expressible in current policy

Fields:

- `signal_id`
- `signal_type`
- `scope`
- `severity`
- `supporting_trace_ids_json`
- `summary`
- `recommended_level`
- `created_at`

### improvement_proposal

The concrete proposed improvement.

Fields:

- `proposal_id`
- `signal_id`
- `level`
- `title`
- `problem_statement`
- `evidence_summary`
- `recommended_change`
- `affected_files_json`
- `risk_level`
- `rollback_plan`
- `status`
- `created_at`
- `approved_by`
- `approved_at`

### change_audit

Immutable audit trail for every step after a proposal exists.

Implemented audit-only events:

- `proposal_created`
- `human_approved`
- `code_change_package_prepared`
- `verification_started`
- `verification_passed`
- `verification_failed`
- `human_review_requested`
- `human_review_approved`
- `human_review_rejected`
- `branch_pushed`
- `mr_created`
- `mr_merged`
- `deployment_plan_prepared`
- `deployment_started`
- `deployment_succeeded`
- `deployment_failed`
- `post_deploy_monitor_recorded`
- `rollback_started`
- `rollback_succeeded`

Future execution-capable implementations may add `branch_created` and
`patch_generated` once branch creation and patch generation are no longer
metadata-only.

Fields:

- `audit_id`
- `proposal_id`
- `event_type`
- `actor`
- `payload_json`
- `created_at`

## Code Improvement Workflow

### 1. Detect improvement need

The supervisor identifies that policy/config cannot express a repeated issue.

Example:

> Workers can emit many tool events for 30 minutes without artifact, state transition, or useful comment. Existing stale detection only sees time, not high-activity low-progress behavior.

### 2. Create proposal

The proposal must include:

- exact traces and outcomes that justify the change
- why config-only change is insufficient
- target detector or action path
- expected behavior before and after
- affected modules
- risk level
- rollback plan

### 3. Human approval gate

The supervisor may prepare a draft plan, but it must not edit source files before approval.

Approval record must include:

- approver identity
- proposal id
- approved level, such as E3 or E4
- allowed target repository
- allowed branch prefix
- verification commands
- deployment target, if E4 is approved

### 4. Isolated branch/worktree

After approval, create an isolated work area.

Required branch naming:

```text
warden/improve-<proposal-id>-<short-slug>
```

The implementation worktree must not be inside production runtime directories unless explicitly approved. For Project Next worktrees, use the machine-local convention outside the repo. For Hermes plugin work, use a stable development checkout rather than mutating the live installed source in place.

### 5. Patch

Each code proposal should normally include:

- detector or policy implementation
- unit tests or fixture tests
- dry-run output update
- docs/examples update
- migration if a state schema change is required

Patch scope must stay tied to the proposal. Avoid opportunistic refactors.

### 6. Verify

Minimum verification for code self-improvement:

- unit tests for the changed detector/action
- idempotency tests when the action path mutates Kanban state
- dry-run before/after fixture showing the new decision
- migration test if state tables change
- lint/type checks where already used by the repo

Verification output is written into `change_audit`.

### 7. Human review

The supervisor presents:

- proposal summary
- evidence traces
- diff summary
- verification output
- risks and rollback
- branch/MR link if already pushed

Human review is required before merge or deployment.

### 8. Commit and push

Commit message format:

```text
feat(warden): add <detector-or-policy> loop improvement

Proposal: <proposal_id>
Evidence: <signal_id>
Verification: <commands>
```

Push behavior:

- push to a proposal branch only
- do not push directly to protected main branches
- create MR/PR when repository integration supports it

Recorded publication results require nonblank branch URL and pull request URL
metadata.

Current implementation records caller-provided merge metadata only. It does not
merge pull requests, update protected branches, or create release artifacts.
Recorded merge results require base branch, merge commit, merger, and timestamp
metadata.

### 9. Deploy/update

For E4, deployment must be explicit and target-scoped.

Current implementation records caller-provided deployment metadata only. It does not
run restart/reload commands, push artifacts, mutate live installs, or execute health
checks.

Current implementation can also prepare a caller-provided deployment plan after an
external merge is recorded. The plan is metadata only and does not mutate runtime
state.
The planned commit must match the recorded merge commit.
Prepared deployment plans require plugin version metadata.
Prepared deployment plans require restart/reload command metadata.
Prepared deployment plans require monitor window metadata.

Deployment result recording requires a prepared deployment plan.
Recorded deployment results must match the prepared plan's target profile,
commit/version, config, restart command, monitor window, and rollback command
metadata.
Recorded deployment results require health check status metadata.

Deployment record includes:

- target profile(s)
- plugin version or commit SHA
- config changes
- restart/reload commands
- health check result
- post-deploy monitor window

### 10. Monitor and rollback

Post-deploy monitor should compare:

- new detector trigger count
- false positive rate
- verification failure rate
- manual override rate
- gateway errors or performance regressions

Current implementation records caller-provided monitor summaries only. It does not
poll gateways, collect metrics, evaluate alert rules, or execute rollback.
Recorded monitor summaries must match the prepared deployment plan's target
profile and monitor window metadata.

Rollback plan must be executable from the audit record:

- revert config
- reinstall prior plugin version or commit
- restart affected gateway services
- mark proposal as rolled back

Current implementation records caller-provided rollback metadata only. It does not
execute rollback commands, reinstall plugins, restart services, or mutate live
profiles.
Recorded rollback results must match the prepared deployment plan's target
profile and rollback command metadata.
Recorded rollback results require restored commit and plugin version metadata.
Recorded rollback results require health check status metadata.

## Example Proposal

```yaml
proposal_id: imp-2026-06-22-high-activity-low-progress
level: E3
title: Detect high-activity low-progress worker loops
problem_statement: >
  Several worker tasks emit many tool events but do not create artifacts,
  state transitions, or useful comments. Existing stale detection waits
  on time thresholds and misses this failure mode.
evidence:
  traces: [trace-101, trace-117, trace-203]
  repeated_pattern: 3
  average_wasted_duration_minutes: 42
recommended_change:
  detector: high_activity_low_progress
  rule: >
    If a running task emits at least 20 events in 30 minutes with no artifact,
    no state transition, and no accepted progress comment, mark no_progress.
risk_level: medium
verification:
  - pytest tests/test_no_progress_detector.py
  - kanban-warden --config fixtures/high-activity.yaml dry-run
rollback_plan: >
  Disable policies.worker_execution.high_activity_low_progress.enabled,
  or revert the plugin commit and restart affected gateway services.
```

## Safety Rules

- Self-improvement must be evidence-backed.
- Evidence must come from durable trace/outcome records, not a single ad hoc observation.
- Config changes are preferred over code changes.
- Code changes require tests.
- Deployment requires post-deploy monitoring.
- Rollback must be defined before deployment.
- Humans approve boundaries; the supervisor executes within approved boundaries.

## Open Questions

- Which repository becomes the canonical active `kanban-warden` development repo instead of the archived checkout?
- Should proposals be stored in the warden SQLite DB, Kanban cards, Git issues, or all three?
- Which approval surface is preferred: Kanban card approval, Feishu approval, GitHub PR review, or Hermes chat confirmation?
- Which profiles are eligible for E4 controlled rollout in the first phase?
- What is the minimum evidence threshold before a proposal can be generated?

## Recommended Rollout

1. Build v0.2 loop trace and post-action verification.
2. Build v0.3 improvement signals and config proposals.
3. Run v0.3 in recommendation-only mode for at least one week.
4. Enable v0.3 config-change proposals for low-risk policy values.
5. Implement v0.4 E3 code proposal generation without deployment.
6. Add E4 controlled rollout only after code proposal quality is proven.
